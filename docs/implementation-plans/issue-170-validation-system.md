# Implementation Plan: Feature Completeness Validation System (Issue #170)

## Overview
Build a validation system to prevent incomplete feature implementations (like empty audio files, broken integrations, incomplete features) by validating implementation tasks against their acceptance criteria before marking them complete.

## KEY ARCHITECTURAL DECISION

**Validation Strategy: Source Code Discovery**

Marcus will validate implementations by:
1. **Getting `project_root`** from workspace manager or logged design artifacts
2. **Scanning `project_root`** to discover source files (*.js, *.py, *.html, etc.)
3. **Reading source code** to verify features match acceptance criteria
4. **NOT relying on agents to log source code** - Marcus discovers files automatically

This approach works because:
- Marcus runs locally and has filesystem access to `project_root`
- Agents create source files in the project workspace
- Artifacts (design docs) are separate from source code
- Marcus can discover what agents built without agents explicitly reporting it

## Revised Evidence Sources (Source Code Discovery)

The validation system will analyze THREE types of evidence:

1. **Source Code Files** - Discovered by scanning `project_root` directory
   - Agent creates files during implementation
   - Marcus discovers them by directory traversal
   - Read file content to verify features

2. **Artifacts (Design Docs)** - Logged via `log_artifact()` in `docs/` subdirectories
   - Specs, API schemas, architecture docs
   - Provide context for what SHOULD be implemented
   - NOT the implementation itself

3. **Decisions** - Logged via `log_decision()`, retrieved from `get_task_context()`
   - Architectural choices
   - Implementation rationale
   - Context for understanding completeness

**What validation actually checks**:
- **Functional completeness**: Are ALL required features implemented in SOURCE CODE?
- **Not half-built**: No placeholder code, no TODOs, no missing integrations
- **Acceptance criteria met**: Each criterion must have evidence in SOURCE FILES
- **Not just empty files**: Source file content must show actual implementation

**Evidence is used to verify**:
- **Source code content** (discovered via directory scan) shows what was implemented
- Code snippets from source files prove features exist (e.g., right-click handler, validation logic)
- Artifacts (design docs) provide context for expected implementation
- Decisions provide rationale for implementation choices
- File sizes catch empty files (0 bytes)

## Entity Roles Clarification

**Worker Agent** (external to Marcus):
- Implements source code in `project_root`
- Reports "completed" when done
- Receives validation feedback or blocker
- Fixes issues and retries
- Can dispute validation (triggers dual LLM voting)

**Marcus** (orchestrator only - context must stay CLEAN):
- Creates blocker in Kanban (for visibility)
- Passes WorkAnalyzer's suggestions to worker
- Does NOT analyze code
- Does NOT provide guidance
- Maintains task coordination context only

**WorkAnalyzer** (validation module with separate LLM):
- Discovers source files (scans `project_root`)
- Reads ENTIRE source files
- Validates functional completeness
- Provides remediation (first failures - tactical code fixes)
- Provides blocker advice (repeated failures - strategic meta-analysis)
- Has dedicated LLM instance (isolated from Marcus's context)

### Remediation vs Blocker Advice (DIFFERENT ANALYSIS)

**Remediation** (first failures - from WorkAnalyzer's validation LLM):
- **Scope**: Tactical code-level fixes
- **Context**: Current validation failure only
- **Format**: Specific actionable fixes
- **Example**:
  ```
  ISSUE: Phone validation not implemented
  REMEDIATION: Add validatePhone() function to validation.js:

  function validatePhone(phone) {
    return /^\d{3}-\d{3}-\d{4}$/.test(phone);
  }
  ```

**Blocker Advice** (repeated failures - from WorkAnalyzer's meta-analysis LLM):
- **Scope**: Strategic meta-analysis of why worker is stuck
- **Context**: FULL validation history + pattern analysis
- **Format**: Different approach recommendations
- **Example**:
  ```
  PATTERN DETECTED: You've attempted validation 3 times with same missing feature.

  ANALYSIS:
  - Attempt 1: validation.js had no validatePhone()
  - Attempt 2: validation.js still had no validatePhone()
  - Attempt 3: validation.js still had no validatePhone()

  YOU ARE STUCK because you keep submitting without fixing the specific issue.

  STRATEGIC ADVICE:
  1. STOP and READ the remediation carefully - it tells you EXACTLY what's missing
  2. The problem is NOT your overall approach - your form HTML is correct
  3. The ONLY issue is: validation.js needs a validatePhone() function
  4. Don't rebuild everything - just add the ONE missing function
  5. If you don't know phone validation regex, copy this exact code:
     function validatePhone(phone) {
       return /^\d{3}-\d{3}-\d{4}$/.test(phone);
     }

  ALTERNATIVE APPROACHES (if above doesn't work):
  - Use HTML5 pattern attribute instead: <input pattern="[0-9]{3}-[0-9]{3}-[0-9]{4}">
  - Use a validation library like validator.js
  - Ask for help understanding why validation.js isn't being updated

  If you're still stuck after trying these approaches, report blocker manually.
  ```

**Key Difference**: Blocker advice analyzes THE PATTERN OF FAILURES (meta-level) instead of just the current failure (tactical).

   - For subtasks: also includes sibling subtask decisions
   - Provides full architectural context for validation
6. Return `WorkEvidence` bundle with:
   - Discovered source files (path, **COMPLETE** content, size, features)
   - Design artifacts (specs/docs for expected implementation)
   - Decisions (rationale from this task + dependencies)

**`validate_implementation_task(task, state)`**:
1. Gather evidence (discover source files + decisions + design artifacts)
2. Build AI prompt with:
   - **Each acceptance criterion individually**
   - **Discovered source code files** (AI can verify features in actual implementation)
   - File sizes (to catch empty files like `audio/click.mp3` = 0 bytes)

   - Design artifacts (expected implementation from specs)
   - Decisions (for context)
   - **Strict instruction**: "Check source code for EACH criterion - if ANY criterion missing, FAIL"
3. Call LLMAbstraction for criterion-by-criterion analysis
4. Parse response into ValidationResult with specific missing features
5. Return pass/fail with actionable remediation

## Key Implementation Details

### Evidence Gathering Strategy (Source Code Discovery)
The WorkAnalyzer will collect THREE types of evidence:

1. **Get project_root**:
   ```python
   # Try workspace manager first
   if hasattr(state, 'workspace_manager') and state.workspace_manager.project_config:
       project_root = state.workspace_manager.project_config.main_workspace
   else:
       # Fall back to extracting from logged artifacts
       artifacts = state.task_artifacts.get(task_id, [])
       if artifacts and 'project_root' in artifacts[0]:
           project_root = artifacts[0]['project_root']
   ```

2. **Discover Source Files** (scan `project_root`):
   ```python
   # Recursively find implementation files
   source_extensions = {'.js', '.py', '.html', '.css', '.java', '.ts', '.tsx', '.jsx'}
   exclude_dirs = {'docs', 'node_modules', '.git', '__pycache__', 'venv', '.venv'}

   source_files = []
   for root, dirs, files in os.walk(project_root):
       # Filter out excluded directories
       dirs[:] = [d for d in dirs if d not in exclude_dirs]

       for file in files:
           if Path(file).suffix in source_extensions:
               file_path = Path(root) / file
               size_bytes = file_path.stat().st_size

               # Read ENTIRE content (safety limit: 1MB)
               if size_bytes == 0:
                   content = ""  # Empty file - validation should FAIL!
               elif size_bytes > 1_000_000:  # 1MB safety limit
                   # File too large - read first 100KB and flag for manual review
                   content = file_path.read_text(errors='ignore')[:100000]
                   content += "\n\n[FILE TRUNCATED - Too large for validation]"
               else:
                   # Read complete file to verify ALL acceptance criteria
                   content = file_path.read_text(errors='ignore')

               # Scan for placeholders
               has_todos = 'TODO' in content or 'FIXME' in content
               has_stubs = 'NotImplementedError' in content or 'pass  # TODO' in content

               source_files.append({
                   'path': str(file_path),
                   'relative_path': str(file_path.relative_to(project_root)),
                   'size_bytes': size_bytes,
                   'content': content,
                   'has_placeholders': has_todos or has_stubs
               })
   ```

3. **Get Design Artifacts & Decisions**:
   ```python
   # Design artifacts (specs/docs for context about WHAT should be built)
   design_artifacts = state.task_artifacts.get(task_id, [])

   # Decisions (WHY agent made certain choices)
   context = await get_task_context(task_id, state)
   decisions = context.get('decisions', [])
   ```

**KEY INSIGHT**: Marcus can validate WITHOUT relying on agents to log source code. Just scan `project_root` to discover what they actually built.

The AI will be instructed to:
- **FAIL immediately** if no source files discovered (empty project)
- **FAIL** if any acceptance criterion has NO code evidence in source files
- **FAIL** if source code has TODOs/placeholders for required features
- **FAIL** if source files are empty (0 bytes) for required components
- **FAIL** if obvious integrations missing (event handlers, API calls, etc.)
- **PASS** only if ALL criteria have working implementation in source code

### Remediation Strategy

**Example: Warranty Form Incomplete Implementation**

Agent creates form but forgets phone validation:
```python
# Git diff shows:
warranty-form.html: <input name="name">, <input name="email">, <input name="phone">
```

## Key Design Clarifications

### 1. File Reading Strategy
**MUST read ENTIRE source files** (not truncated) to verify ALL acceptance criteria:
- Empty files (0 bytes): Read as empty string → FAIL validation
- Normal files (<1MB): Read completely to check all features
- Large files (>1MB): Read first 100KB + flag for manual review (edge case)
- **Why complete files**: A criterion might be implemented at line 500 - truncation would miss it

### 2. Decision Context Scope
`get_task_context()` provides **comprehensive architectural context**:
- Decisions from **this task** (implementation choices)
- Decisions from **dependency tasks** (design phase decisions, API contracts)
- For subtasks: decisions from **sibling subtasks** (parallel work)
- **Why this matters**: Validation can see "why" agent made choices, not just "what" they built
- Example: Design task decides "use client-side validation" → implementation task implements it → validator sees both decision AND implementation

### 3. Failure & Blocker Workflow
**No automatic retry limit** - agent-driven recovery:

**Validation Failure → Agent Retry Loop**:
1. Marcus returns validation failure with **specific remediation** (task stays IN_PROGRESS)
2. Agent reads failure response, fixes issues in source code
3. Agent reports "completed" again (retry)
4. Marcus validates again:
   - **Different issues** or **PASS**: Normal workflow continues
   - **Same exact issues**: Agent stuck → Marcus creates blocker via `report_blocker()`

**After Blocker Created**:
5. `report_blocker()` runs **AI analysis** and provides **actionable suggestions**
6. Agent reads blocker with AI suggestions
7. Agent decides:
   - Try AI suggestions (blocker remains, agent keeps working)
   - Manually report blocker if truly stuck (agent gives up)
8. **No abandonment**: Task stays assigned until agent succeeds or explicitly reports blocker

**Key Insight**: Marcus creates blocker **only when agent is stuck in loop with same issues** - this prevents infinite retry but respects agent autonomy to try different approaches.
